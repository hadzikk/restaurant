# Check and make validation before changes

## Purpose
- To make sure incoming changes relevant with the goals
- To decrease technical debt in the future
- To get know what this project really needs
- To confirm at first before making any changes and make recommendation for changes

## Input
- Check the supabase schema.sql and understand it's database design
- Check with actual code is, it well implemented the database design relatively to schema.sql?
- Check is the code following best practices convention?
- Check is it following Modular Monolith with DDD principle?

## Guardrails
- Don't change the SQL Schema
- Don't create additional prompts in prompts/ folder

## Expected output
- Clarify your understanding about this project
- Summarise your analysis
- Make recommendation
